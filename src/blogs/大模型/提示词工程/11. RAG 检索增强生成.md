# Retrieval Augmented Generation (RAG)

RAG（检索增强生成）是目前大语言模型（LLM）落地应用中最热门的技术架构之一。本指南将带您从概念到实战，全面了解 RAG。

## 概念

**定义**：RAG（Retrieval-Augmented Generation）是一种将**信息检索**（Retrieval）与**文本生成**（Generation）相结合的技术架构。

**通俗类比——"开卷考试"**：

- **传统的 LLM（如 ChatGPT）**：就像学生参加**闭卷考试**。通过预训练（学习），模型记住了很多知识，但如果遇到没学过的（最新的新闻）或私有的（公司内部文档）知识，它可能会答错或瞎编（幻觉）。

- **RAG 架构**：就像学生参加**开卷考试**。当遇到问题时，学生（LLM）先去翻阅参考书（检索外部数据库），找到相关段落（Context），然后结合书本内容和自己的理解来回答问题。

## 原理

RAG 的运作流程通常分为三个步骤：

1. **检索 (Retrieve)**：当用户提出问题时，系统首先在外部知识库（通常是向量数据库）中搜索与问题最相关的文档片段。

2. **增强 (Augment)**：将检索到的相关信息（Context）与用户的原始问题（Prompt）通过特定的模版拼接在一起，形成一个包含丰富背景信息的提示词。

3. **生成 (Generate)**：将这个增强后的提示词输入给 LLM，模型根据提供的上下文生成准确的回答。

**为什么能提升效果？**

RAG 能够显著提升 LLM 的回答质量，主要原因如下：

1. **减少幻觉 (Hallucination)**：限制模型仅基于检索到的事实进行回答，而不是依赖其可能模糊的"内部记忆"，从而大幅降低胡编乱造的概率。

2. **获取私有数据**：通用的 LLM 无法训练到你公司内部的 PDF、Wiki 或数据库。RAG 允许模型"外挂"这些私有数据，而无需重新训练模型。

3. **知识实时更新**：重新训练一个大模型需要数周时间和昂贵成本。RAG 只需要更新外部知识库（例如插入一条新新闻），模型就能立刻回答最新问题。

4. **可解释性与溯源**：RAG 生成的答案可以标注引用的来源文档（如："参考自《员工手册》第 3 页"），这在法律、医疗等严肃场景下至关重要。

## Prompt 模版

在 RAG 架构中，提示词工程（Prompt Engineering）是连接检索结果与生成模型的桥梁。

**标准 RAG Prompt 结构**：

```
[角色设定]
你是一个智能助手。请严格根据提供的[上下文信息]来回答用户的[问题]。

[约束条件]
- 如果上下文中没有包含答案，请直接回答"根据提供的信息，我无法回答这个问题"，不要试图编造。
- 回答要简洁明了。

### 上下文信息 (Context)：
{context_chunks}

### 用户问题 (Question)：
{user_query}

### 你的回答：
```

### 举例

假设我们有一个关于"光速科技公司报销制度"的知识库。

**你的输入（构造后的 Prompt）：**

```
你是一个行政助手。请根据以下上下文回答员工的问题。如果文中未提及，请回答不知道。

### 上下文信息：
1. 【差旅费】员工出差期间的餐饮补贴标准为每天 200 元。
2. 【交通费】市内交通推荐使用网约车，实报实销。
3. 【审批流】所有超过 1000 元的报销都需要部门总监审批，1000 元以下由直属经理审批。
4. 【发票】必须提供增值税专用发票。

### 用户问题：
我出差吃了一顿饭花了 150 元，这笔钱需要总监审批吗？
```

**AI 输出：**

```
不需要总监审批。

根据上下文信息：
1. 餐饮补贴标准为每天 200 元，您的花费 150 元在标准内。
2. 关于审批流，只有超过 1000 元的报销才需要部门总监审批，1000 元以下由直属经理审批。

因此，您的 150 元报销仅需直属经理审批。
```

### 使用技巧

1. **"不知道"指令 (Negative Constraints)**：务必在 Prompt 中明确指示："如果上下文中找不到答案，请说不知道"。这能防止模型利用自带的（可能是错误的或过时的）训练数据进行回答。

2. **引用来源 (Citation)**：要求模型在每一句陈述后加上引用标记。例如：`请在回答中引用上下文的编号，如 [1]`。这有助于用户验证信息的真实性。

3. **上下文排序 (Re-ranking)**：检索出的片段可能有多个，通常将相关性最高的片段放在 Prompt 的**最前面**或**最后面**（利用 LLM 的"首尾效应"），效果通常比放在中间好。

4. **思维链 (Chain of Thought)**：对于复杂问题，可以在 Prompt 中加入 `请一步步思考`，引导模型先分析检索到的上下文，再得出结论。

## 补充知识

### 1. RAG vs 微调 (Fine-tuning)

很多初学者容易混淆这两者，以下是对比：

| 特性 | RAG (检索增强生成) | Fine-tuning (微调) |
|------|-------------------|-------------------|
| **比喻** | 参加开卷考试（带课本） | 考前突击复习（背课本） |
| **知识更新** | 实时（更新数据库即可） | 慢（需要重新训练） |
| **数据隐私** | 数据在外部，可灵活控制权限 | 数据内化在模型参数中，难以分割 |
| **主要用途** | 获取特定事实、私有知识查询 | 学习特定的语言风格、格式或行业术语 |
| **成本** | 较低（推理成本 + 数据库） | 较高（训练算力 + 数据清洗） |

**结论**：
- 如果你的目标是让 AI 知道"公司上个月的销售额"或"最新的产品手册"，请选 **RAG**；
- 如果你的目标是让 AI 学会像"鲁迅"一样说话，或学会特定的 JSON 输出格式，请选 **微调**。

### 2. 核心技术栈 (Tech Stack)

如果你想动手实现一个 RAG 系统，通常需要以下工具：

- **向量数据库 (Vector Database)**：用于存储和检索知识。
  - _常用工具_：Chroma, Weaviate, Milvus, Pinecone

- **Embedding 模型**：将文字转化为向量（数字列表），以便计算相似度。
  - _常用模型_：OpenAI text-embedding-3, HuggingFace M3E / BGE (中文效果佳)

- **编排框架 (Orchestration Framework)**：将检索和生成串联起来的胶水代码。
  - _核心工具_：**LangChain**, **LlamaIndex**

- **LLM**：负责最后生成的"大脑"。
  - _选项_：GPT-4o, Claude 3.5 Sonnet, DeepSeek, Qwen (通义千问)

### 3. RAG 优化进阶技巧

| 优化方向 | 具体方法 | 效果 |
|---------|---------|------|
| **分块策略** | 按语义/段落分块，而非固定长度 | 提升检索相关性 |
| **混合检索** | 向量检索 + 关键词检索结合 | 提高召回率 |
| **重排序 (Reranking)** | 使用专门的排序模型对检索结果二次排序 | 提升精度 |
| **查询重写** | 对用户问题进行扩展或改写 | 改善检索效果 |
| **多跳检索** | 针对复杂问题进行多轮检索 | 解决复杂问答 |

### 4. 常见陷阱与应对

| 陷阱 | 表现 | 应对方法 |
|------|------|---------|
| **上下文过长** | 检索内容超出模型上下文限制 | 采用分层摘要或迭代压缩 |
| **检索噪声** | 检索到不相关内容干扰回答 | 提高相似度阈值，使用重排序 |
| **信息冲突** | 多个来源信息矛盾 | 在 Prompt 中要求模型指出冲突并说明 |
| **过度依赖检索** | 对常识问题也依赖检索 | 设计混合策略，简单问题直接回答 |

> **实践建议**：RAG 是当前 LLM 落地的最佳实践之一。成功的 RAG 系统需要迭代优化检索和生成两个环节，建议从简单方案开始，逐步引入高级优化技术。
