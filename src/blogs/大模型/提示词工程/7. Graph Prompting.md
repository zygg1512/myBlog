# Graph Prompting (图提示)

Graph Prompting（图提示）是一种前沿的提示工程技术，它不仅是让 AI "一步步思考"，而是引导 AI 将问题建模为**图（Graph）结构**——包含节点（Nodes）和边（Edges）。这种方法特别适用于处理非线性、多对多关系或需要全局视角的复杂推理任务。

## 概念

**Graph Prompting** 的核心在于将思维过程或数据结构化为一张网络图，而不是一条线（如 Chain of Thought）或一棵树（如 Tree of Thoughts）。

- **节点 (Nodes)**：代表核心概念、实体、步骤或独立的思维片段。
- **边 (Edges)**：代表节点之间的关系（因果、关联、顺序、对比等）。

通过这种方式，AI 不再是单向地从 A 到 B，而是可以在多个概念之间通过"跳跃"或"聚合"来寻找答案，模拟人类的网状思维。

## 原理

为什么将任务"图化"能提升效果？

1. **拓扑感知 (Topology Awareness)**：大语言模型虽然是序列生成（一个字接一个字），但它在训练数据中学习到了丰富的结构化知识。显式地要求模型构建图，能激活其对复杂拓扑关系的处理能力。

2. **非线性推理 (Non-linear Reasoning)**：现实世界的问题往往不是线性的（A导致B，B导致C）。很多时候是循环的（A影响B，B反过来影响A）或者多对多的。Graph Prompting 允许模型在推理时考虑这种复杂的相互作用。

3. **信息聚合 (Information Aggregation)**：在图中，一个节点可以连接多个其他节点。模型可以被引导去"汇总"所有相邻节点的信息来得出一个更全面的结论，避免管中窥豹。

## 举例

假设我们需要分析一个复杂的商业案例，找出导致公司股价下跌的**核心循环原因**，而不仅仅是表面原因。

### 场景：分析 TechCorp 公司的危机

**你的输入：**

```
请使用 Graph Prompting 的思路分析以下文本。
第一步：提取文本中的关键事件作为"节点"。
第二步：分析事件之间的因果关系作为"边"。
第三步：基于构建的图，找出导致危机的关键"恶性循环"路径。

[文本]：
TechCorp 为了短期财报好看，削减了 30% 的研发预算。这导致核心产品迭代延迟，用户开始抱怨功能陈旧。因为用户体验下降，口碑变差，导致新用户增长率暴跌。为了挽回营收，市场部提高了广告投放成本。高昂的广告费加上营收下降，导致季度利润进一步缩水。管理层看到利润缩水，决定继续削减各部门预算，包括研发部。
```

**AI 输出：**

```
### 1. 构建图结构

**节点 (Nodes):**
N1: 削减研发预算
N2: 产品迭代延迟
N3: 用户体验下降/口碑变差
N4: 新用户增长率暴跌
N5: 营收下降
N6: 提高广告投放成本
N7: 季度利润缩水

**边 (Edges/因果关系):**
N1 -> N2 (削减预算导致迭代延迟)
N2 -> N3 (迭代延迟导致体验下降)
N3 -> N4 (体验差导致增长暴跌)
N4 -> N5 (增长暴跌导致营收下降)
N6 -> N7 (广告费高导致利润缩水)
N5 -> N7 (营收下降导致利润缩水)
N7 -> N1 (利润缩水导致管理层继续削减预算)

### 2. 识别关键路径与循环

**识别到的恶性循环 (Cycle):**
N1 -> N2 -> N3 -> N4 -> N5 -> N7 -> N1

**分析结论:**
TechCorp 陷入了一个典型的"削减成本致死螺旋"。
起始点虽然是 N1 (削减预算)，但通过一系列传导（N7 → N1）形成了一个闭环。仅仅解决市场投放 (N6) 无法打破这个循环，必须在 N1 (恢复研发投入) 或 N3 (提升用户体验) 处进行干预，切断反馈回路。
```

## 使用技巧

1. **显式定义符号系统**：在提示词中约定一种清晰的图表示法，例如 Mermaid 语法、DOT 语言，或者简单的 `[A] --> [B]` 格式。这能让 AI 生成的结果更容易被解析和理解。
   - _技巧_："请用 Mermaid 格式输出你的思维图。"

2. **分步图遍历 (Step-by-step Traversal)**：不要让 AI 一口气输出结论。先让它列出节点，再列出关系，最后再基于图进行推理。
   - _Prompt_："先列出所有涉及的人物（节点），再列出他们之间的互动（边），最后基于这个网络找出谁是中心人物。"

3. **用于"头脑风暴"与"知识图谱"**：当你需要发散思维时，Graph Prompting 非常好用。
   - _Prompt_："将'气候变化'作为中心节点，向外扩展两层相关的经济与社会影响节点。"

4. **自我修正图**：构建完初步的图后，可以要求 AI："检查图中是否有遗漏的连接？是否存在逻辑矛盾的边？"

## 提示模式对比

理解 Graph Prompting 的最佳方式是将其与其他模式进行对比：

| 模式 | 结构 | 核心逻辑 | 适用场景 |
|------|------|---------|---------|
| **Chain of Thought (CoT)** | 线形 (Line) | A → B → C | 数学题、逻辑推理、步骤明确的任务。 |
| **Tree of Thoughts (ToT)** | 树形 (Tree) | A → B1/B2 → C | 需要探索多种可能性、决策树、写大纲。 |
| **Graph Prompting** | 网状 (Graph) | A ↔ B, A → C, C → B | **因果循环分析、复杂叙事梳理、多维知识关联、代码依赖分析。** |

## 常用图表示法

### 1. 简单箭头表示法
```
节点A --> 节点B [关系描述]
节点B --> 节点C [关系描述]
```

### 2. Mermaid 语法（推荐）
```
graph TD
    A[削减研发预算] --> B[产品迭代延迟]
    B --> C[用户体验下降]
    C --> D[新用户增长暴跌]
    D --> E[营收下降]
    E --> F[利润缩水]
    F --> A
```

### 3. 邻接矩阵表示
适用于节点数量较多、关系复杂的场景：
```
       N1   N2   N3   N4
N1     0    1    0    0
N2     0    0    1    0
N3     0    0    0    1
N4     1    0    0    0
```

## 应用场景详解

| 场景 | 节点示例 | 边示例 | 分析目标 |
|------|---------|--------|---------|
| **故障排查** | 系统组件、日志事件 | 依赖关系、调用链 | 找出故障传播路径 |
| **叙事分析** | 小说人物、关键事件 | 人物关系、因果关系 | 识别剧情漏洞或关键转折 |
| **知识图谱** | 实体（人、地点、概念） | 语义关系 | 发现隐藏关联 |
| **代码分析** | 函数、类、模块 | 调用关系、依赖关系 | 识别循环依赖、重构机会 |
| **组织架构** | 部门、人员 | 汇报关系、协作关系 | 找出沟通瓶颈 |

## 局限性与注意事项

### 局限性
1. **规模限制**：当节点数量过多（>20个）时，模型可能难以处理复杂的网络关系
2. **关系歧义**：边的关系定义可能不够精确，导致理解偏差
3. **静态视角**：难以表示随时间动态变化的图结构

### 应对策略
- **分层抽象**：将大图拆分为多个子图，逐层分析
- **明确关系类型**：定义关系词汇表（如"导致"、"依赖"、"包含"）
- **时间切片**：对于动态图，按时间阶段分别构建图

> **实践建议**：Graph Prompting 适合作为分析工具而非生成工具。在复杂问题分析、根因定位等场景下，它能帮助梳理思路、发现隐藏的关联和循环。
