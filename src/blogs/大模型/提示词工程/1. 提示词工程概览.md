# 提示词工程概览

## 核心定义：什么是提示词工程？

提示词工程是用自然语言对非确定性（Stochastic）大语言模型（LLM）进行"编程"的过程。

它的核心目标是通过设计特定的上下文、指令和结构，约束 LLM 的发散性，引导其**稳定、准确地输出符合系统要求的结果**（如严格的 JSON 结构、特定的代码逻辑或推理结论），从而将大模型无缝接入软件业务流中。

## 工程落地的边界：不仅仅是"写提示词"

在真实的业务系统或自动化评估平台中，提示词工程远不止在聊天框中打字，它包含一整套需要通过代码（如 Node.js）落地的系统化开发工作：

1. **动态上下文管理：** 提示词通常是动态组装的模板。需要通过代码从数据库或外部工具动态拉取业务数据（如日志、代码片段），并安全地注入到 Prompt 模板中。
2. **API 调度与流式控制：** 处理与 LLM 之间的网络请求、身份验证、重试机制以及 SSE（Server-Sent Events）流式数据传输。
3. **输出解析与兜底机制：** 编写正则或解析器，从模型返回的非结构化自然语言中强制提取结构化数据（如 JSON），并在解析失败时触发重试或备选逻辑。
4. **自动化评估流水线：** 结合评估工具构建自动化测试流，通过自定义断言（Assertions）批量计算准确率、召回率等指标，量化评估 Prompt 修改后的实际效果。

## 常见提示词技术分级体系

技术复杂度由浅入深，主要分为四个层级：

### Level 1：基础构建 (Foundational) - "直来直去"

适用于日常交互和简单、常见的单步任务。

- **Zero-shot Prompting (零样本提示)**
  - **概念：** 直接下达指令，不提供任何参考示例。依赖模型预训练知识。
  - **应用：** 简单的文本翻译、日常对话、基础常识问答、情感倾向初步判断。
  - **局限性：** 对复杂任务或特定格式要求难以精准控制。

- **Few-shot Prompting (少样本提示)**
  - **概念：** 在指令前提供几个"输入→输出"的正确样例，让模型进行模式匹配。
  - **应用：** 强制模型输出特定的 JSON/XML 格式、统一客服机器人的回复语调、特定垂直领域的小样本意图分类。
  - **局限性：** 示例质量直接影响输出质量，需要精心设计的样例。

### Level 2：逻辑与推理 (Reasoning & Logic) - "教模型思考"

适用于涉及数学计算、复杂逻辑或需要严谨推导的任务。

- **Chain-of-Thought Prompting (思维链 / CoT)**
  - **概念：** 引导模型展现思考的中间过程（如"请一步一步地思考"），降低复杂逻辑任务的幻觉率。
  - **应用：** 逻辑推理题、应用数学题解析、简单的业务规则推导。
  - **局限性：** 会增加输出token数量，提升成本；对简单任务可能"过度思考"。

- **Self-Consistency (自一致性 / 自洽性)**
  - **概念：** 对同一问题生成多条不同的推理路径，最终采用"多数表决（Majority Vote）"选出出现频率最高的答案。
  - **应用：** 高要求的金融数据核对、医疗诊断初步筛选、事实性问答的防幻觉校验（结合评估平台做并发测试）。
  - **局限性：** 需要多次调用模型，成本成倍增加；仅适用于有明确答案的问题。

- **Tree of Thoughts (思维树 / ToT)**
  - **概念：** 思维链的进化版。将推理过程扩展为一棵树，允许模型探索多条分支，并结合搜索算法（如 BFS/DFS）进行"回溯"和全局评估。
  - **应用：** 极其复杂的任务规划（如多并发系统架构设计）、长篇小说的多线剧情生成、需要试错的数学证明（如 24 点游戏求解）。
  - **局限性：** 实现复杂度高，需要自定义搜索策略；计算成本较高。

- **Multimodal CoT (多模态思维链)**
  - **概念：** 将思维链扩展到视觉/音频领域。模型结合输入的图像特征进行分步推理。
  - **应用：** 工业图纸的自动缺陷分析、几何数学题（读图解题）、财务报表（图表截图）的深度逻辑分析。
  - **局限性：** 依赖多模态模型能力；图像理解的准确性受限于模型训练数据。

- **Graph Prompting (图提示)**
  - **概念：** 把复杂问题和背景知识转化为图（Graph）结构（节点和边），指导模型理解实体之间复杂的网络关系来进行关联推理。
  - **应用：** 知识图谱问答、复杂的供应链风险传递分析、医疗病例中多种并发症的关联推理。
  - **局限性：** 需要预先构建或提取图结构；对非结构化数据的处理能力有限。

### Level 3：结构与工程流水线 (Structure & Pipeline) - "模块化架构"

此类技术通常不再是单一的文本，而是需要 Node.js 等后端代码进行调度拼接的工程架构。

- **Prompt Chaining (提示链)**
  - **概念：** 将庞大的任务拆解为多个低耦合的小型 Prompt 节点，前一个节点的输出作为后一个节点的输入。
  - **应用：** 自动化代码审查流水线（节点A找Bug → 节点B写修复代码 → 节点C生成PR说明）、长篇报告的自动化组装。
  - **局限性：** 节点间错误会累积传播；整体延迟增加；需要设计合理的中间状态格式。

- **Generate Knowledge Prompting (生成知识提示法)**
  - **概念：** 两步走策略：先引导模型生成与问题相关的"背景知识"，再将生成的知识作为上下文进行最终作答。
  - **应用：** 缺乏领域常识的冷门知识问答、规避模型在回答反直觉问题时产生的常识性幻觉。
  - **局限性：** 生成的知识可能有误，导致错误答案；需要额外的调用步骤。

- **Meta Prompting (元提示)**
  - **概念：** 直接向模型定义解题的"语法"、"结构"和"逻辑框架"，让模型将其视为通用模版。
  - **应用：** 构建跨领域的通用分析模版（如通用的"SWOT分析提示词引擎"），或者定义代码生成的通用架构模式，而非具体实现。
  - **局限性：** 需要模型具备较强的抽象理解能力；框架设计不当会影响效果。

- **Retrieval Augmented Generation (检索增强生成 / RAG)**
  - **概念：** 在生成回答前，先用代码从外部数据库（如向量数据库）中检索出相关的真实信息片段，将其注入上下文。
  - **应用：** 解决企业私有数据问答（如"根据最新的员工手册，我的年假有多少天"）、智能客服知识库、实时新闻问答。
  - **局限性：** 检索质量决定生成质量；对检索不到的冷门问题表现可能下降；需要维护向量数据库。

- **Automatic Prompt Engineer (自动提示词工程师 / APE)**
  - **概念：** 使用一个大模型来自动生成、评估和优化另一个大模型的提示词。
  - **应用：** AI 评估平台的核心功能。开发者输入目标和测试数据，系统自动迭代跑出准确率最高的 Prompt，极大节省人工调优时间。
  - **局限性：** 需要大量标注数据进行评估；自动生成的提示词可能可解释性较差。

- **Active-Prompt (主动提示)**
  - **概念：** 批量处理数据时，计算模型对不同问题的不确定度，主动挑出模型最"拿不准"的几个问题交给人工作答，然后将人工回答作为 Few-shot 示例。
  - **应用：** 高精度的垂直领域数据集标注（如法律合同的关键条款提取），把人类专家的精力用在刀刃（边缘 Case）上。
  - **局限性：** 需要设计合理的不确定性度量方法；需要人工介入。

- **Directional Stimulus Prompting (定向刺激提示)**
  - **概念：** 引入一个独立的"刺激词"或"线索"（Hint），作为强引导，迫使模型沿着特定的方向生成结果。
  - **应用：** 强制包含特定关键词的 SEO 文章生成、确保不偏离主旨的长文本摘要、引导模型提取特定视角的关键信息。
  - **局限性：** 过度引导可能导致生成内容生硬或缺乏多样性。

### Level 4：代理与动作 (Agentic & Action) - "让模型长出手脚"

前沿的工程应用，涉及 LLM 与外部系统（API、执行环境、MCP 协议等）的深度交互。

- **ReAct (推理与行动)**
  - **概念：** 引导模型在"思考（Thought）"与"行动（Action）"之间交替循环，根据观察结果决定下一步调用什么工具或输出结论。
  - **应用：** 构建全能型 AI Agent。例如结合 MCP 协议，让大模型自主决定何时读取本地 Git 仓库、何时搜索网页、何时写入文件。
  - **局限性：** 需要定义清晰的工具接口；循环次数可能失控；错误累积风险。

- **Automatic Reasoning and Tool-use (自动推理与工具使用 / ART)**
  - **概念：** ReAct 的进阶版，模型能够"零样本"地自动从庞大的工具库中挑选组合工具，并生成对应的推理步骤。
  - **应用：** 应对未知的复杂任务环境。例如系统突然接入了 50 个新的 API，Agent 能够不依赖人工编写示例，自动阅读 API 文档并组合调用它们来完成数据分析。
  - **局限性：** 工具选择错误风险；对工具描述的质量要求高；执行链路长导致延迟高。

- **Program-Aided Language Models (程序辅助语言模型 / PAL)**
  - **概念：** 遇到复杂的数学或逻辑计算时，让模型写一段代码（如 Python/Node.js），由外部执行器运行，将运行结果作为答案。
  - **应用：** 复杂的财务利息计算、精准的时间/日期推导、需要严谨物理公式的工程计算（彻底解决 LLM 算术不准的致命弱点）。
  - **局限性：** 需要安全的代码执行环境；代码可能有bug；不适合非计算类任务。

- **Reflexion (反射 / 自我反思)**
  - **概念：** 当模型调用工具失败或输出错误时，它会自动生成一段"反思（Reflection）"记录下来，存入短期/长期记忆中，确保在下一次尝试时避开同样的坑。
  - **应用：** 自动化软件测试与修复（写测试 → 失败 → 反思 → 修改代码的内循环）、长期运行的自主智能体（如 24 小时监控并修复服务器异常的运维 Agent）。
  - **局限性：** 反思质量依赖于模型能力；需要记忆管理机制；可能陷入反思循环。

## 技术选型速查表

| 技术 | 实现复杂度 | 调用成本 | 适用场景 | 关键考量 |
|------|-----------|---------|---------|---------|
| Zero-shot | ⭐ | 低 | 简单任务、快速原型 | 任务简单、无需精确控制 |
| Few-shot | ⭐ | 低 | 格式控制、语调统一 | 需要高质量示例 |
| CoT | ⭐⭐ | 中 | 逻辑推理、数学问题 | 增加输出长度 |
| Self-Consistency | ⭐⭐ | 高 | 高精度要求 | 多次调用成本 |
| ToT | ⭐⭐⭐ | 高 | 复杂规划、创意生成 | 需要自定义搜索策略 |
| Prompt Chaining | ⭐⭐⭐ | 中 | 多步骤任务 | 错误传播风险 |
| RAG | ⭐⭐⭐ | 中 | 私有数据问答 | 需要维护知识库 |
| ReAct | ⭐⭐⭐⭐ | 中高 | 工具调用Agent | 需要工具接口设计 |
| PAL | ⭐⭐ | 中 | 精确计算 | 需要安全执行环境 |

> **选型建议**：从 Level 1 开始尝试，只有在遇到明确瓶颈时才升级到更高层级。高复杂度技术带来更高成本和更多失败模式，不要为了"先进"而过度设计。
