# Shot Prompting 样本提示

## 零样本提示 (Zero-shot Prompting)

### 概念

**零样本提示**是指在向 AI 下达指令时，**不提供任何完成该任务的参考示例**。你直接描述你需要什么，并期望 AI 能够理解并直接给出正确的输出。

### 原理

零样本提示能够奏效，依赖于大语言模型（LLM）的**预训练 (Pre-training) 和泛化能力**。

由于模型在训练阶段已经"阅读"了互联网上的海量文本（如维基百科、书籍、新闻等），它已经内化了人类语言的基本规则、逻辑推理能力和大量的常识。因此，对于常规任务，它不需要临时看例子，就能通过"直觉"和积累的"知识"直接解决问题。

### 举例

你直接抛出任务，不给任何参考格式：

**你的输入 (Prompt)：**
```txt
提取以下文段中的核心人物和地点：

"公元前221年，秦始皇在咸阳宫举行了盛大的庆典，宣告天下的统一。"
```

**AI 输出：**
```txt
核心人物：秦始皇
地点：咸阳宫
```

## 少样本提示 (Few-shot Prompting)

### 概念

**少样本提示**是指在向 AI 提出最终问题之前，在提示词中**先提供少量（通常是 1 到 5 个）相关的"问题-答案"示例**。这些示例作为参考模板，引导 AI 按照特定的思路、风格或格式来回答最后一个问题。

### 原理

少样本提示的核心原理是**上下文学习 (In-context Learning)**。

大语言模型本质上是强大的"模式匹配器"。当你在提示词中提供示例时，AI 会在当前的对话上下文中瞬间捕捉到这些示例之间的**内在规律**（比如特定的排版格式、特殊的逻辑推理路径、或是某种幽默语气）。AI 会将这种规律直接套用在最后一个无答案的问题上，从而极大地提高输出的准确性和规范性。

### 举例

当你需要一种不常见的、非常严格的输出格式时，通过给出示例来"教" AI：

**你的输入 (Prompt)：**
```txt
请提取文本信息，并严格按照字典格式输出：
文本：张三昨天在北京买了一部手机。
输出：{"Name": "张三", "Location": "北京", "Item": "手机"}

文本：李四打算下周去上海看演唱会。
输出：{"Name": "李四", "Location": "上海", "Item": "演唱会"}

文本：王五刚刚在广州吃了一碗美味的烧鹅饭。
输出：
```

**AI 输出：**
```txt
{"Name": "王五", "Location": "广州", "Item": "烧鹅饭"}
```

### 示例数量的选择原则

示例数量并非越多越好，需要根据任务复杂度权衡：

| 示例数量 | 适用场景 | 成本考量 |
|---------|---------|---------|
| **1个** | 简单格式模仿、单一模式任务 | token消耗最小 |
| **2-3个** | 分类任务（覆盖正负样本）、中等复杂度格式 | 性价比最佳 |
| **4-5个** | 复杂推理模式、多类别分类 | 效果显著提升 |
| **5个以上** | 上下文长度允许的极少数复杂场景 | 边际收益递减，谨慎使用 |

> **注意**：示例数量增加会线性增加 token 消耗和推理成本，同时可能占用更多上下文窗口空间。

### 使用技巧

- **格式必须高度一致：** 你的示例格式（标点符号、换行、前缀）应该与你最后提问的格式一模一样。AI 是非常敏感的模式匹配器。

- **示例要具有代表性：** 如果你要做情感分类，提供的样本里最好既包含正面，也包含负面和中性，让 AI 看到全貌。示例之间尽量相互独立且完全穷尽（**MECE 原则**：Mutually Exclusive, Collectively Exhaustive，即相互独立、完全穷尽）。

- **不要给错误示例：** 除非你在进行极其高级的反面教材提示，否则给 AI 的样本答案必须是完全准确的，因为 AI 会盲目相信你提供的"标准答案"。

- **从 Zero 开始，再向 Few 升级：** 遇到新任务时，先试着用 Zero-shot 直接问。如果效果不好、格式不对，再添加样本升级为 Few-shot。这可以节省你的时间。

- **示例顺序的影响：** 研究表明，示例的排列顺序可能对输出产生影响。建议将最清晰、最典型的示例放在前面，或随机排列多组测试。

### 常见陷阱与规避方法

| 陷阱 | 表现 | 规避方法 |
|------|------|---------|
| **标签泄露** | 示例中的标签无意中透露了答案规律 | 确保示例分布与真实场景一致，避免过度简化 |
| **格式不一致** | 示例间格式不统一导致模型困惑 | 使用模板化方式生成示例，确保严格一致 |
| **示例偏见** | 示例覆盖不全面导致输出偏见 | 遵循 MECE 原则，确保各类情况都有代表 |
| **过度拟合** | 模型过度模仿示例表面形式而忽略任务本质 | 增加示例多样性，或结合指令明确任务目标 |

### 实战案例：优化前后对比

**任务**：将用户评论按情感分类（正面/负面/中性）

**❌ Zero-shot（效果不佳）：**
```txt
请分析以下评论的情感倾向：
"这个产品的包装还可以，但是用起来感觉一般般。"
```
AI 可能输出冗长的分析段落，而非简洁的分类标签。

**✅ Few-shot（效果显著）：**
```txt
请分析评论的情感倾向（正面/负面/中性）：

评论：这个手机太棒了，强烈推荐！
情感：正面

评论：完全不值这个价，退货了。
情感：负面

评论：昨天收到的，还在试用中。
情感：中性

评论：这个产品的包装还可以，但是用起来感觉一般般。
情感：
```
AI 输出：`中性` —— 精准、简洁、可直接入库。

### 与 Zero-shot Prompting 的区别

| **特性** | **零样本提示 (Zero-shot)** | **少样本提示 (Few-shot)** |
|---------|---------------------------|--------------------------|
| **提供示例** | 0 个 | 1 到多个 (通常 2-5 个) |
| **适用场景** | 常规任务（翻译、总结、常识问答） | 复杂推理、特定格式输出、风格模仿、分类任务 |
| **编写成本** | 低（随手即写） | 较高（需要精心挑选具有代表性的示例） |
| **AI 准确度** | 依赖模型本身的预训练知识 | 显著提高，因为有了参考标准 |
| **token 成本** | 低 | 随示例数量增加而增加 |

> **选型建议**：所有任务都应先尝试 Zero-shot，当输出格式不稳定或结果不准确时，再引入 Few-shot。这是最高效的提示词开发流程。
