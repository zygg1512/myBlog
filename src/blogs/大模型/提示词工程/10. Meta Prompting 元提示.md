# Meta Prompting (元提示)

> 注：本指南基于 PromptingGuide.ai 及相关学术研究（如 Zhang et al. 2024, Reynolds et al. 2024）的定义。

Meta Prompting 是一种专注于任务结构（Structure）和语法（Syntax）的高级提示技术。与传统的关注"内容"的提示方法不同，Meta Prompting 旨在通过抽象的框架来引导大语言模型（LLM）。

## 概念

**定义**：Meta Prompting 是一种通过提供**抽象的任务结构**、**语法规则**或**通用模式**来指导模型解决问题的技术。它不依赖具体的、特定的例子（如 Few-Shot），而是教给模型一套"解题的元规则"或"思维框架"。

**通俗类比——"数学公式 vs 例题"**：

- **Few-Shot Prompting (传统)**：就像给学生看几道做好的**例题**（1+1=2, 2+2=4），让学生照猫画虎。如果遇到差异很大的题，学生可能不会做。

- **Meta Prompting**：就像直接教给学生**数学公式**（a + b = c）和**运算定律**。学生学会的是解题的"结构"，因此能处理更多未见过的题型。

## 原理

Meta Prompting 的核心在于 **"结构优于内容" (Structure over Content)**。

1. **抽象化 (Abstraction)**：将具体的任务细节剥离，提炼出任务的通用逻辑。

2. **语法与类型 (Syntax & Types)**：利用编程语言风格、伪代码或特定的格式标签，强制模型按照严谨的逻辑路径思考。

3. **零样本/抽象少样本 (Zero-shot / Abstract Few-shot)**：即使不提供真实数据的例子，通过提供"抽象示例"（例如用 `<X>` 代替具体数字），模型也能理解任务的处理流程。

**为什么能提升效果？**

1. **避免偏差**：传统的 Few-Shot 可能会让模型过分关注例子中的特定词汇（Recency Bias），导致在新任务中生搬硬套。Meta Prompting 只有骨架，没有干扰内容。

2. **增强推理**：通过强制规定输出的句法结构（如 JSON、伪代码），迫使模型进行更严密的逻辑推理。

3. **通用性**：同一个 Meta Prompt 模版可以应用于同一类别的所有问题，无需为每个细分问题寻找新的例子。

## Prompt 模版

Meta Prompt 通常包含：**任务定义** + **抽象结构示例** + **输入槽位**。

**通用结构**：

```
[任务描述]
你是一个[角色]。你的任务是根据[抽象规则]处理输入。

[抽象模式/语法]
请遵循以下逻辑结构进行回答：
输入类型: <描述输入的数据类型>
逻辑步骤 1: <描述第一步操作，而非具体内容>
逻辑步骤 2: <描述第二步操作>
输出格式: <定义的输出格式>

[抽象示例] (关键：这里不给真实数据，只给变量占位符)
Example:
输入: [实体 A] 与 [实体 B] 发生交互
思考: 基于规则分析 A 和 B 的关系。
输出: 关系(A, B)

[真实输入]
{用户问题}
```

### 举例

假设我们要让 AI 判断句子的逻辑谬误类型。我们使用 Meta Prompting 提供的**抽象结构**，而不是给具体句子的例子（避免模型模仿句子的语气）。

**你的输入：**

```
指令：识别文本中的逻辑谬误。
请使用以下【抽象结构】进行分析，不要直接给出结论。

结构 (Structure):
1. 主张 (Claim): <提取核心观点>
2. 前提 (Premise): <识别支持该主张的前提>
3. 谬误模式 (Flaw Pattern): <匹配已知的谬误模式，如人身攻击、稻草人谬误、滑坡谬误等>
4. 结论 (Conclusion): <谬误名称>

抽象示例 (Abstract Example - 请勿使用真实内容):
1. 主张: X 是正确的。
2. 前提: 人物 Y 说 X 是错误的，但 Y 具有 [某种负面属性]。
3. 谬误模式: 攻击提出观点的人（Y），而不是攻击观点本身（X）。
4. 结论: 人身攻击 (Ad Hominem)

真实输入 (Real Input):
"你不能相信老张关于公司管理的建议，因为他昨天吃饭居然逃单了。"
```

**AI 输出：**

```
1. 主张: 老张关于公司管理的建议不可信。
2. 前提: 老张昨天吃饭逃单了（个人品德问题）。
3. 谬误模式: 攻击老张的个人品德（逃单），但这与他的管理专业能力没有直接逻辑关系。
4. 结论: 人身攻击 (Ad Hominem)
```

## 使用技巧

1. **使用伪代码或格式标记**：在 Prompt 中引入类似编程的结构（如 `if-then`，XML 标签），能显著提升模型遵循指令的稳定性。
   - _Bad_: "先想一下，然后告诉我..."
   - _Good_: `Step 1: <Analyze>; Step 2: <Conclusion>...`

2. **定义"类型" (Type Definition)**：像写代码一样定义输入数据的"类型"。例如："输入是一段包含日期的文本"，而不是直接扔一段文本进去。这能激活模型的分类处理能力。

3. **保持"内容中立"**：在编写 Prompt 模版时，尽量不要包含任何带有强烈情感或特定领域知识的词汇，除非那是任务必须的。保持模版的纯净度。

## Meta Prompting 的其他定义

值得注意的是，AI 领域发展迅速，"Meta Prompting" 这个词有时也被用于指代其他相关技术：

- **Agentic Orchestration (智能体编排)**：指模型将任务拆解并分发给不同"专家"的过程。这在某些论文（如 Suzuki et al.）中也被称为 Meta-Prompting。
- **Prompt Generation (提示词生成)**：指用 AI 帮你写 Prompt (即 APE)，有时在通俗语境下也被混用为 Meta Prompting。

## 与 Few-Shot 的对比

| 维度 | Few-Shot Prompting | Meta Prompting |
|------|-------------------|----------------|
| **核心** | 具体示例 | 抽象结构 |
| **泛化能力** | 受示例限制 | 跨任务通用 |
| **偏差风险** | 高（模仿表面形式） | 低（理解深层结构） |
| **编写难度** | 中（需找好示例） | 高（需抽象能力） |
| **适用场景** | 格式模仿、风格迁移 | 逻辑分析、结构化任务 |

## 应用场景详解

| 场景 | Meta Prompt 设计要点 |
|------|---------------------|
| **代码审查** | 定义检查步骤（安全性、性能、可读性）而非给出具体代码示例 |
| **文本分类** | 定义分类维度（情感、主题、紧急度）和判断标准 |
| **数据提取** | 定义字段类型、格式规则、验证逻辑 |
| **逻辑分析** | 定义分析框架（主张-前提-结论） |
| **创意生成** | 定义创意维度（新颖性、可行性、受众匹配度） |

## 最佳实践

### 设计高质量 Meta Prompt 的步骤

1. **分析任务本质**：剥离具体案例，找出通用模式
2. **定义输入输出类型**：明确数据的结构和约束
3. **设计处理步骤**：用抽象语言描述转换逻辑
4. **提供抽象示例**：用占位符展示结构，而非真实数据
5. **验证通用性**：用不同类型的输入测试模版是否都能工作

### 常见陷阱

| 陷阱 | 表现 | 应对 |
|------|------|------|
| **过度抽象** | 结构过于通用，失去指导意义 | 在抽象与具体之间找到平衡 |
| **结构冲突** | 定义的结构与模型预训练不匹配 | 参考模型擅长的格式（如 JSON、Markdown） |
| **缺失验证** | 没有约束条件的结构容易被忽略 | 添加明确的验证规则或检查点 |

## 与 APE 的关系

Meta Prompting 与 **Automatic Prompt Engineer (APE)** 有协同效应：

- **Meta Prompting** 提供结构框架
- **APE** 可以自动优化该框架的具体措辞

例如：
1. 人工设计 Meta Prompt 的结构
2. 使用 APE 自动调整每个步骤的表述，找到最优版本

> **实践建议**：Meta Prompting 适合需要严格结构化输出的场景（如数据提取、逻辑分析、代码审查）。它的前期设计成本较高，但一旦模版成型，可以复用于同类任务，具有很高的投入产出比。
