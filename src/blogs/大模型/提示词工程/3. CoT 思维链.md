# Chain-of-Thought Prompting (思维链 / CoT)

Chain-of-Thought（思维链，简称 CoT）是提示工程中最经典、最实用的技术之一。它通过引导大语言模型（LLM）展示中间推理步骤，从而显著提升模型解决复杂数学、逻辑和常识推理问题的能力。

## 概念

**Chain of Thought (CoT)** 的核心在于打破"输入 → 输出"的黑盒模式，将其转化为"输入 → **推理步骤** → 输出"的透明模式。

简单来说，就是要求 AI **"把解题过程写出来"**，而不是直接猜答案。

- **Standard Prompting (标准提示)**：直接问结果。
  - *问*：罗杰有5个网球，又买了2罐，每罐3个。他现在有几个？
  - *AI 答*：11个。（或者错误的答案）

- **CoT Prompting (思维链提示)**：要求展示过程。
  - *问*：...（同上）
  - *AI 答*：罗杰一开始有5个球。2罐网球，每罐3个，所以是 $2 \times 3 = 6$ 个。$5 + 6 = 11$。所以答案是11个。

## 原理

CoT 的本质是将一个复杂的计算或推理任务，拆解为一系列简单的线性步骤。

**为什么能提升效果？**

1. **分配更多计算量 (Compute Allocation)**：LLM 生成每个 token 都需要计算时间。直接生成答案可能导致"计算时间"不足。强制生成推理步骤，相当于给了模型更多的"思考时间"和计算资源来处理复杂逻辑。

2. **引导逻辑流 (Logical Guidance)**：推理步骤充当了"脚手架"。步骤 A 的输出成为步骤 B 的输入，模型可以基于自己生成的中间结论继续推导，降低了跳跃性思维带来的错误。

3. **可解释性与调试 (Debuggability)**：如果 AI 答错了，我们可以通过检查思维链，发现是在哪一步逻辑出了问题（是算错了，还是理解错了），从而优化提示词。

## 举例

假设我们要解决一个稍有难度的逻辑推理题。

**你的输入：**

```
去食堂买饭。
苹果 3 元一个，香蕉 2 元一个。
小明买了 3 个苹果，小红买了 2 个香蕉。
他们一共付了 20 元，找回多少钱？

请一步步思考（Let's think step by step）。
```

**AI 输出：**

```
我们一步步来计算：

1.  **计算小明花的钱**：
    小明买了 3 个苹果，每个 3 元。
    花费 = 3 × 3 = 9 元。

2.  **计算小红花的钱**：
    小红买了 2 个香蕉，每个 2 元。
    花费 = 2 × 2 = 4 元。

3.  **计算总花费**：
    总花费 = 9 + 4 = 13 元。

4.  **计算找回的钱**：
    他们一共付了 20 元。
    找回 = 20 - 13 = 7 元。

答案：找回 7 元。
```

> **注意**：如果不使用 CoT，模型在处理更复杂的数字混合文本时，很容易直接给出错误的数字。

## 使用技巧

### 1. Zero-Shot CoT (零样本思维链)

这是最简单但也极其有效的方法。只需在 Prompt 结尾加上一句魔法咒语：

> **"Let's think step by step."** (让我们一步步思考) 或者中文版："请一步步进行逻辑推理。"

### 2. Few-Shot CoT (少样本思维链)

在 Prompt 中提供 1-3 个包含完整推理过程的范例（Example）：

```
问题：一个农场有 5 只鸡，每只鸡每天下 2 个蛋。一周能收集多少个蛋？
推理：农场有 5 只鸡，每只鸡每天下 2 个蛋，所以每天产蛋 5 × 2 = 10 个。一周有 7 天，所以一周产蛋 10 × 7 = 70 个。
答案：70

问题：书店有 120 本书，周一卖出 1/4，周二卖出剩余的一半，还剩多少本？
推理：周一卖出 120 × 1/4 = 30 本，剩余 120 - 30 = 90 本。周二卖出 90 ÷ 2 = 45 本，最后剩余 90 - 45 = 45 本。
答案：45

问题：[你的新问题]
推理：
```

这种方式效果通常优于 Zero-Shot，因为它规范了输出格式和推理深度。

### 3. 指定输出结构

为了方便程序提取，可以明确要求结构化输出：

```
请先在 <thinking> 标签中写出推理过程，然后在 <answer> 标签中写出最终答案。
```

### 4. 推理步骤格式设计

设计良好的推理格式能显著提升效果：

- **编号步骤**：使用"1. 2. 3."清晰标记每个推理阶段
- **显式计算**：要求模型写出中间计算过程，而非跳步
- **结论先行或殿后**：根据下游任务决定最终答案位置

## CoT 适用与不适用场景

不是所有任务都需要 CoT。滥用 CoT 会增加 Token 消耗并导致回答啰嗦。

| 适用场景 (推荐使用 CoT) | 不适用场景 (不推荐) |
|------------------------|-------------------|
| **数学应用题** (Math Word Problems) | **简单常识问答** (如"法国首都是哪里？") |
| **逻辑推理** (Logic Puzzles) | **翻译任务** (通常直接翻译即可) |
| **代码生成与解释** (Coding) | **创意写作** (小说、诗歌) |
| **多步骤规划** (Agent Planning) | **情感闲聊** |
| **复杂决策分析** | **简单分类任务** |

## CoT 的局限性与注意事项

### 1. Token 成本增加

CoT 会显著增加输出 token 数量（通常增加 3-10 倍），直接推高调用成本。在高并发场景下需要谨慎评估成本收益比。

### 2. 对简单任务可能产生反效果

对于简单任务，强制模型"一步步思考"可能导致：
- **过度思考**：简单问题复杂化，降低响应速度
- **引入幻觉**：推理步骤中可能产生无根据的假设

### 3. 推理错误累积

如果中间步骤出错，后续步骤会基于错误继续推导，导致最终答案必然错误。这也是为什么 CoT 常与 Self-Consistency 结合使用。

## 高级变体：Auto-CoT

**Auto-CoT (Automatic Chain-of-Thought)** 是一种自动化生成 Few-shot CoT 示例的方法。

这种技术通过一句简单的 "Let's think step by step" 作为提示，来激发大语言模型自动生成推理过程，从而帮你自动拼凑出 Few-shot 的参考示例。整个过程分两步走：

1. **Question Clustering (问题聚类)**：先把题库里各种各样的问题进行分类，划分成不同的 Clusters，确保选出来的题目不重样，保证示例的多样性。
2. **Demonstration Sampling (示例采样)**：从每个分类里挑出一道有代表性的题目，然后直接用 Zero-Shot-CoT 让模型自己写出解题步骤。为了防止模型啰嗦或跑题，系统会加上一些简单的 Heuristics（启发式规则），比如限制题目的 Token 长度（例如不超过 60 个 Token），或者规定 Rationale（推导过程）最多只能写 5 步。这样就能逼着模型交出最精简、最准确的“标准答案”来作为演示。


**优势**：省去人工编写示例的工作量，且示例多样性更好。

## 组合使用建议

CoT 与其他技术结合效果更佳：

| 组合技术 | 效果提升 | 适用场景 |
|---------|---------|---------|
| **CoT + Self-Consistency** | 显著降低推理错误 | 高精度数学/逻辑任务 |
| **CoT + Few-shot** | 规范推理格式 | 需要结构化输出的场景 |
| **CoT + 输出格式约束** | 便于下游解析 | 自动化流水线 |

> **实践建议**：在生产环境中，建议先用 Zero-Shot CoT 测试效果，若准确率不达标，再升级为 Few-Shot CoT，必要时结合 Self-Consistency 进行多轮投票。

## CoT 失效的常见原因排查

当 CoT 效果不佳时，检查以下方面：

1. **任务本身是否适合分解**？某些直觉型任务难以拆解为步骤
2. **提示词是否明确要求推理**？"Let's think step by step"的位置和措辞很重要
3. **模型能力是否足够**？较小模型（< 7B）可能难以生成高质量推理链
4. **示例质量是否过关**？Few-Shot CoT 的示例必须逻辑严密、格式统一
