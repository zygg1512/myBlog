# Generate Knowledge Prompting (生成知识提示)

## 概念

**Generate Knowledge Prompting (生成知识提示)** 是一种提示工程技术，旨在解决大语言模型（LLM）在面对需要特定背景知识或常识推理的问题时，因"回忆"不起相关知识而导致回答错误的问题。

它的核心流程是：**在要求模型回答问题之前，先明确要求模型生成与该问题相关的知识或事实。**

简单来说，就是让 AI "先复习一下相关知识点"，然后再做题，而不是直接凭直觉猜答案。

## 原理

这种方法的有效性建立在以下原理之上：

1. **知识唤醒 (Knowledge Retrieval)**：LLM 内部存储了海量知识，但在直接回答问题（尤其是常识性或推理偏见较强的问题）时，可能无法立即激活正确的知识路径。

2. **上下文增强 (Contextual Priming)**：通过强制模型先输出相关知识，这些知识就变成了当前对话的"上下文"的一部分。模型在生成最终答案时，会关注并利用这些刚刚生成的、高质量的上下文信息。

3. **减少幻觉**：当模型基于明确列出的事实（即使是它自己生成的）进行推理时，比直接生成答案更符合逻辑，减少了胡编乱造的可能性。

## 举例

假设我们需要回答一个涉及常识物理属性的问题。如果直接问，模型可能会忽略物体大小的细节。

你的输入：

```
请按照以下步骤回答问题：
1. 生成关于问题中涉及物体的相关知识（如大小、物理状态、用途等）。
2. 基于生成的知识，回答最后的问题。

问题：高尔夫球能装进普通的红酒杯吗？
```

AI 输出：

```
**知识生成：**
1. **高尔夫球**：是一个硬质球体，标准直径约为 42.67 毫米（约 4.3 厘米）。它通常用于户外运动，体积较小。
2. **红酒杯**：通常由玻璃制成，用于盛装葡萄酒。红酒杯的形状多样，但标准的红酒杯口径通常在 6 厘米到 10 厘米之间，杯肚最宽处通常更宽。
3. **空间关系**：如果物体 A 的直径小于容器 B 的开口直径和内部容纳空间，那么物体 A 可以放入容器 B。

**最终回答：**
是的，高尔夫球能装进普通的红酒杯。根据上述知识，高尔夫球的直径（约 4.3 厘米）明显小于普通红酒杯的开口直径（通常大于 6 厘米），因此可以轻松放入。
```

## 使用技巧

要高效使用 Generate Knowledge Prompting，建议遵循以下模板：

### 1. 显式步骤法

在 Prompt 中明确拆分指令，强迫模型"慢思考"。

- _话术：_ "在回答这个问题之前，请先列出 3 条与此话题相关的核心事实。"

### 2. 多角度知识生成 (Diversity)

对于复杂问题，单一知识点可能不够。要求模型从不同角度生成知识。

- _话术：_ "请先从经济、环境和政治三个角度生成关于'电动车普及'的背景知识，然后回答它对传统能源行业的影响。"

### 3. 知识整合 (Integration)

生成知识后，要明确指示模型**使用**这些知识。

- _话术：_ "基于上述生成的知识，解释为什么……"

### 4. 这里的知识必须是"确定的"

这种方法最适合用于**常识推理**（Commonsense Reasoning）或**事实性问题**。如果问题本身是开放性的创意写作，这种方法效果不明显。

## 与 RAG 的区别

虽然 Generate Knowledge Prompting 和 **RAG (检索增强生成)** 看起来很像（都是先找知识再回答），但有本质区别：

| 特性 | Generate Knowledge Prompting | RAG (Retrieval Augmented Generation) |
|------|-----------------------------|-------------------------------------|
| **知识来源** | **模型内部**（模型自己写出来的） | **外部数据库**（搜索引擎、文档库） |
| **适用场景** | 常识推理、逻辑推导、通用知识 | 私有数据查询、最新时事、极度精确的细节 |
| **成本/速度** | 低（无需外部工具，只消耗 token） | 高（需要检索系统支持） |
| **知识准确性** | 依赖模型训练数据，可能有幻觉 | 依赖外部数据源，可更新、可追溯 |
| **实现复杂度** | 低（纯 Prompt 工程） | 高（需要向量数据库、嵌入模型等） |

**总结：** 当你没有外部数据库，但希望模型回答得更严谨时，使用 Generate Knowledge Prompting 是性价比最高的选择。

## 适用场景详解

| 场景类型 | 示例 | 生成知识的方向 |
|---------|------|---------------|
| **物理常识** | "这个物体能浮在水上吗？" | 物体密度、材质特性 |
| **生物知识** | "企鹅为什么不会飞？" | 生理结构、进化适应 |
| **历史背景** | "这个历史事件的影响是什么？" | 时间线、相关人物、社会背景 |
| **技术概念** | "微服务架构适合这个项目吗？" | 架构特点、适用场景、优缺点 |
| **数学问题** | "这个方程有几个解？" | 方程类型、判别条件、特殊 cases |

## 失效场景与应对

### GKP 可能失效的情况

1. **模型本身缺乏相关知识**
   - 表现：生成的知识是错误的
   - 应对：改用 RAG 从外部知识库检索

2. **知识过于专业或冷门**
   - 表现：生成的知识过于笼统或错误
   - 应对：使用 Few-shot 提供领域内的示例

3. **需要实时信息**
   - 表现：模型生成的是过时知识
   - 应对：必须使用 RAG 或工具调用获取最新信息

## 进阶技巧：两阶段优化

对于复杂问题，可以扩展为两阶段生成：

**阶段一：广泛生成**
```
请列出与"[主题]"相关的 10 个关键事实或概念。
```

**阶段二：筛选聚焦**
```
从上述知识中，筛选出与"[具体问题]"最相关的 3-5 条，然后回答该问题。
```

这种方法特别适合：
- 主题范围较广的问题
- 需要区分主次因素的分析任务
- 避免知识过载导致的推理混乱

## 与 CoT 的结合使用

Generate Knowledge Prompting 可以与 Chain-of-Thought 结合，形成**Knowledge-Augmented CoT**：

```
请按以下步骤回答：
1. 生成与问题相关的基础知识
2. 基于这些知识，一步步推理
3. 给出最终答案

问题：为什么北极熊的皮毛是透明的而不是白色的？
```

这种组合的优势：
- 知识生成为推理提供"原材料"
- CoT 确保推理过程逻辑严密
- 两者互补，显著提升复杂问答的准确性

> **实践建议**：Generate Knowledge Prompting 是"轻量级"的知识增强方案。在没有条件部署 RAG 系统时，它能显著提升模型在常识推理任务上的表现，是成本与效果的理想平衡点。
