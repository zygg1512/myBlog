# Self-Consistency (自一致性)

Self-Consistency 是 Google Research 在 2022 年提出的一种高级 Prompt Engineering 技术，旨在显著提升大语言模型（LLM）在复杂推理任务中的准确率。

## 概念

简单来说，**Self-Consistency（自一致性）** 是"Chain of Thought（思维链）"的进阶版策略。

传统的生成方式是**贪婪解码（Greedy Decoding）**，即让模型生成一个它认为概率最高的答案。而 Self-Consistency 的核心策略是：**"少想一次不如多想几次，最后少数服从多数"。**

它并不直接要求模型给出一个答案，而是让模型针对同一个问题，利用不同的推理路径生成多个回答，然后从这些回答中选出出现频率最高（最一致）的那个作为最终结果。

## 原理

Self-Consistency 的运作流程可以概括为三个步骤：

1. **Prompt 构建**：使用包含思维链（CoT）的 Prompt。
2. **多样化采样**：设置温度（Temperature > 0），让模型对同一个 Prompt 生成多个不同的推理过程（Paths）。
3. **多数投票（Majority Voting）**：统计所有生成结果中的最终答案，选出票数最多的答案。

**为什么能提升效果？**

1. **模拟人类思维**：就像人类在解决难题时会反复检查或尝试不同的解法一样，如果多种不同的思考路径都指向同一个答案，那么这个答案正确的概率就极高。

2. **容错率高**：模型在进行长链条推理时，某一步骤容易出现随机错误（幻觉或计算失误）。单次生成如果错了就是错了，但多次生成可以将这些偶然的随机错误通过统计学方法"过滤"掉。

3. **利用概率空间**：复杂的推理问题通常只有唯一的正确答案，但却有无数种错误的答案。错误往往是发散的（错得五花八门），而正确是收敛的（殊途同归）。

## Prompt 模版

Self-Consistency 的 Prompt 本身与 Chain of Thought (CoT) 的 Few-shot 模版非常相似，其核心区别在于**执行层面**。

**通用公式：** `[Few-Shot 示例 (问题 + 推理过程 + 答案)]` + `[新问题]` + `[触发推理的指令]`

### 举例

假设我们要解决一个数学应用题。

你的输入：

（注意：此 Input 会被发送给模型多次，例如 3 次或 5 次）

```
Q: 罗杰有 5 个网球。他又买了两筒网球，每筒有 3 个。他又丢了 2 个。现在他有多少个网球？
A: 罗杰开始有 5 个。买了两筒，每筒 3 个，所以买了 2 * 3 = 6 个。现在他有 5 + 6 = 11 个。丢了 2 个，所以现在是 11 - 2 = 9 个。答案是 9。

Q: 食堂原本有 23 个苹果。如果你用掉 20 个做午餐，然后又买了 6 个。现在有多少个苹果？
A: 原本 23 个。用掉 20 个，剩 23 - 20 = 3 个。买了 6 个，所以 3 + 6 = 9 个。答案是 9。

Q: 一个杂技演员有 10 个球。他丢掉了 3 个，然后又买了一盒球，这盒球的数量是他现在手里球数量的 2 倍。他现在总共有多少个球？
A:
```

AI 输出：

由于我们要进行 Self-Consistency，我们需要模型输出多次（假设采样 3 次）：

**输出 1 (Path 1):**

```
杂技演员开始有 10 个球。
丢掉 3 个，剩下 10 - 3 = 7 个。
买了一盒球，数量是手里的 2 倍，也就是 7 * 2 = 14 个。
总共有 7 (手里) + 14 (新买) = 21 个。
答案是 21。
```

**输出 2 (Path 2):**

```
开始有 10 个。
丢了 3 个，10 - 3 = 7。
买了一盒，数量是 7 的 2 倍，即 14 个。
现在他有 7 + 14 = 21 个。
答案是 21。
```

**输出 3 (Path 3 - 假设这是一次错误的推理):**

```
开始 10 个。丢了 3 个变成 7 个。
买了一盒是现在的 2 倍，所以买了 14 个。
只要算新买的就行了，所以是 14 个。
答案是 14。
```

**【最终决策】**

- Path 1 答案: 21
- Path 2 答案: 21
- Path 3 答案: 14
- **统计结果**: 21 (2票), 14 (1票)
- **最终输出**: 21

## 使用技巧

### 1. 调整 Temperature (温度)

- **关键点**：千万不能设为 0。如果 Temperature=0，模型每次输出都一样，Self-Consistency 就失效了。
- **建议值**：通常设置在 `0.5` 到 `0.7` 之间，以确保推理路径具有多样性（Diversity），但又不至于太离谱。

### 2. 采样次数 (Sample Size)

| 采样次数 | 效果提升 | 成本倍数 | 适用场景 |
|---------|---------|---------|---------|
| 3-5 次 | 基础提升 | 3-5x | 资源受限、低延迟要求 |
| 10 次 | 显著提升 | 10x | 平衡方案（推荐） |
| 20-40 次 | 边际提升 | 20-40x | 极高精度要求 |

> **数据参考**：根据 Google Research 论文，在数学推理任务上，Self-Consistency 可将 GPT-3 的准确率从 17.1% 提升至 35.2%（采样 40 次）。

### 3. 答案提取与标准化

在 Prompt 结尾强制要求输出格式，方便通过程序自动提取答案进行投票统计：

```
请严格按照以下格式输出：
<thinking>
[你的推理过程]
</thinking>
<answer>
[最终答案，只能是一个数字/单词]
</answer>
```

### 4. 实现策略：并行 vs 串行

**串行调用**：
```python
answers = []
for i in range(n_samples):
    response = llm.generate(prompt, temperature=0.7)
    answers.append(extract_answer(response))
final_answer = majority_vote(answers)
```
- 优点：实现简单，不占用额外并发资源
- 缺点：延迟随采样次数线性增加

**并行调用**：
```python
import asyncio

async def generate_one():
    return await llm.generate_async(prompt, temperature=0.7)

responses = await asyncio.gather(*[generate_one() for _ in range(n_samples)])
answers = [extract_answer(r) for r in responses]
final_answer = majority_vote(answers)
```
- 优点：延迟低（接近单次调用）
- 缺点：需要并发支持，瞬时资源占用高

### 5. 适用场景

- ✅ **最适合**：数学问题、逻辑推理、符号推理、代码生成（看哪段代码能跑通）
- ❌ **不适合**：创意写作、开放式问答（如"写一首诗"），因为这些任务没有唯一的正确答案

## 投票策略进阶

### 简单多数投票 (Majority Voting)
选择出现频率最高的答案。适合大多数场景。

### 加权投票 (Weighted Voting)
根据推理链的质量赋予不同权重：
- 推理步骤更详细的答案权重更高
- 与问题相似度更高的推理路径权重更高

### 置信度加权
如果模型输出包含置信度分数，可以结合置信度进行加权投票。

## Self-Consistency 的优缺点对比

为了更全面地理解，以下是该方法的局限性与代价：

| 维度 | 说明 |
|------|------|
| **计算成本** | **高**。因为你需要对同一个问题调用 API 多次（例如 10 次），这意味着 Token 消耗和费用会直接翻 10 倍。 |
| **响应延迟** | **高**。串行生成需要等待多次；并行生成需要高并发支持，整体吞吐量要求高。 |
| **系统性错误** | 如果模型对某个概念完全理解错误，它可能会"固执"地在所有路径中都给出同样的错误答案，此时投票机制无法修正错误。 |

## 资源受限时的优化策略

当成本或延迟成为瓶颈时，可考虑：

1. **早期停止 (Early Stopping)**：当某个答案的票数超过阈值（如超过 50%）时提前结束采样
2. **分层采样**：先用低成本小模型进行初步筛选，再用大模型对难分样本进行 Self-Consistency
3. **动态采样数**：根据问题难度动态调整采样次数（简单问题 3 次，复杂问题 10 次）

## 进阶变体

- **Complexity-based Consistency**：不只看票数，还要看推理链的长度或复杂性，倾向于相信推导过程更详细的答案。
- **Self-Refine**：在投票后，让模型对结果进行自我反思和修正。
- **DiVeRSe**：使用不同的 Prompt 变体（而不仅是 Temperature）来生成更多样化的推理路径。

> **实践建议**：在生产环境中，建议先用 5-10 次采样的 Self-Consistency 建立基准，再根据准确率要求和成本预算进行调整。对于关键业务场景，可结合自动化评估平台持续监控效果。
